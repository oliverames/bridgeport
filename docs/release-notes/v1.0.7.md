# Bridgeport 1.0.7

Bridgeport 1.0.7 fixes silent Apple Events denial for AppleScript-based connectors and adds launch-at-login for the menu bar app. The fixes were root-caused during the 2026-07-20 home-server deployment, where the apple-notes connector could never obtain Notes automation consent.

## Automation consent (TCC)

- Declares `NSAppleEventsUsageDescription` in the app bundle Info.plist for both the development bundle (`script/build_and_run.sh`) and the release bundle (`script/package_release.sh`). Without the usage string, macOS refuses Apple Events from connector subprocesses without ever showing a consent prompt.
- Changes daemon installation (CLI `--daemon-install` and the in-app Start Daemon action) to point the LaunchAgent at the app bundle binary when Bridgeport runs from an `.app` bundle, instead of copying a bare binary to `~/.config/bridgeport/bin`. TCC attributes automation consent to the signed bundle, so prompts now appear and grants persist. Bare binaries built with `swift build` keep the previous copy behavior.

## Menu bar app

- Adds a **Launch Bridgeport at login** toggle to the Dashboard's Service group, backed by `SMAppService.mainApp`. The daemon and Cloudflare tunnel LaunchAgents start at login regardless; the toggle governs only the menu bar UI.

## Upgrade notes

- Existing installations whose LaunchAgent points at `~/.config/bridgeport/bin/bridgeport` should rerun `--daemon-install` from the installed app bundle binary (`/Applications/Bridgeport.app/Contents/MacOS/bridgeport --daemon-install`) so the daemon runs from the bundle and can request automation consent.
